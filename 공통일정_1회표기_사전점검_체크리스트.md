## 체크리스트 5항목

### ✅ 항목 1: 데이터 소스 중복 점검

| 구분 | 내용 |
|------|------|
| **점검 항목명** | 동일 이벤트가 API/DB 응답에 2건 이상 존재하는지 |
| **점검 방법** | ① `GET /api/events` 응답 JSON을 `event_id` 기준으로 그룹화하여 건수 카운트 ② Supabase `events` 테이블에서 `id` 컬럼 `GROUP BY id HAVING COUNT(*) > 1` 쿼리 실행 ③ SQL 스니펫: `SELECT id, COUNT(*) FROM events GROUP BY id HAVING COUNT(*) > 1;` |
| **합격 기준** | 동일 ID 일정이 단일 건만 반환됨 (0건 중복) |
| **불합격 기준** | 동일 ID가 2건 이상 반환됨 → DB 레벨 dedupe 필요 |
| **검증 스니펫** | API 응답 배열에서 `new Set(ids).size === ids.length` 검증 |

### ✅ 항목 2: 렌더링 중복 점검

| 구분 | 내용 |
|------|------|
| **점검 항목명** | 동일 공통일정이 서로 다른 카드/영역에 중복 표기되는지 |
| **점검 방법** | ① `events/index.html` 렌더링 완료 후 `document.querySelectorAll('[data-start][data-end]')` 로 전체 카드 수집 ② 수집된 카드 배열에서 `(data-start, data-end, .ev-title textContent)` 3-tuple 기준으로 중복 여부 확인 ③ 콘솔 스니펫 실행: 각 카드의 식별 키를 Set에 누적하여 size 비교 |
| **합격 기준** | 공통일정이 단일 카드에만 렌더링됨 (동일 3-tuple 카드가 1개만 존재) |
| **불합격 기준** | 동일 식별 키를 가진 카드가 2개 이상 DOM에 존재함 |
| **검증 스니펫** | 브라우저 콘솔에서 `Array.from(cards).map(c => c.dataset.start+c.dataset.end+c.querySelector('.ev-title')?.textContent)` 결과의 중복 검사 |

### ✅ 항목 3: data-start/data-end 기간 중복 점검

| 구분 | 내용 |
|------|------|
| **점검 항목명** | 4/5 카드에 적용된 data-start/data-end 속성이 교집합 구간에서 중복 노출시키는지 |
| **점검 방법** | ① 모든 `[data-start][data-end]` 카드의 기간 구간을 파싱하여 쌍별 교집합 여부 확인 ② 교집합이 존재하더라도 동일 이벤트가 아니면 합격, 동일 이벤트가 교집합 구간에 복수 표기되면 불합격 ③ JS 스니펫: 각 카드의 start/end 타임스탬프 쌍을 비교하여 동일 이벤트의 기간 겹침 탐지 |
| **합격 기준** | 동일 일정이 기간 겹침으로 복수 표기되지 않음 |
| **불합격 기준** | 동일 일정의 기간 교집합 구간에서 2건 이상 렌더링됨 |
| **검증 스니펫** | 카드 배열에서 `new Date(start)` ~ `new Date(end)` 구간을 비교하는 O(n²) 교집합 검출 함수 실행 |

### ✅ 항목 4: DOM 삽입 누적 점검

| 구분 | 내용 |
|------|------|
| **점검 항목명** | JS 자동 전환 로직이 re-render 시 기존 노드를 삭제 후 삽입하는지 |
| **점검 방법** | ① `updateEventStatuses()` 함수(60초 setInterval)가 기존 DOM 노드를 제거 없이 append만 하는지 코드 리뷰 ② 브라우저 콘솔에서 `setInterval` 트리거 전후로 `document.querySelectorAll('.ev-card').length` 비교 ③ 동일 이벤트 ID를 가진 `.ev-card` 노드가 항상 1개인지 10회 연속 렌더 사이클 후 확인 |
| **합격 기준** | 동일 이벤트 DOM 노드가 항상 1개만 존재함 (re-render 후에도 노드 수 불변) |
| **불합격 기준** | re-render 시마다 노드가 누적 증가함 (2→3→4…) |
| **검증 스니펫** | `MutationObserver`로 `#events` 컨테이너의 childList 변화를 감시하여 추가/삭제 패턴 확인 |

### ✅ 항목 5: 상태/캐시 초기화 점검

| 구분 | 내용 |
|------|------|
| **점검 항목명** | 이벤트 상태 전환(예정→진행→종료) 시 이전 상태 잔여 표기가 없는지 |
| **점검 방법** | ① `status-dot` 클래스 전환 로직(`soon` → 기본 → `ended`)에서 이전 클래스가 완전히 제거되는지 코드 리뷰 ② 종료된 이벤트가 `ended-events-container`로 이동 시 원래 위치의 DOM 노드가 완전히 제거되는지 확인 ③ `data-movedToEnded` 플래그와 함께 이전 컨테이너에 잔여 노드가 없는지 브라우저 콘솔에서 검증 |
| **합격 기준** | 상태 전환 후 동일 일정이 이전 상태와 현재 상태에 동시 표기되지 않음 (이전 위치 DOM 노드 수 = 0) |
| **불합격 기준** | 상태 전환 후 이전 상태 위치와 현재 상태 위치에 동일 이벤트가 각각 1개씩 총 2개 DOM 노드로 존재함 |
| **검증 스니펫** | `document.querySelectorAll('.ev-card')`와 `document.querySelectorAll('#ended-events-container .ev-card')` 간에 동일 이벤트 ID 노드가 양쪽에 존재하는지 확인 |
